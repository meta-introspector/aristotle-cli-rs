# SplitDecls.lean — Status Report

## Date: 2026-06-20

## Overview

The kernel API splitter (`SplitDecls.lean`) uses the Lean 4 `Environment` API to
split a Lean project into one-file-per-declaration with exact dependency tracking.
This replaces the old regex-based `static_split.py` approach.

The splitter correctly computes dependency graphs and filters builtins/auto-generated
declarations, but **the split files are NOT compilable** because the pretty-printed
kernel expressions don't survive a round-trip through source syntax.

---

## What Works

### Dependency Tracking (100% correct)
- `collectRefs` — walks `Expr` AST collecting all constant references via `Expr.foldConsts`
- `constDeps` — gets dependencies from both type AND value of a constant
- `bfsClosure` — BFS from root declarations to find transitive dependency closure
- `topoSort` — iterative DFS post-order traversal, O(V+E), handles cycles
- `filterDeps` — restricts deps to the project closure (not all 471K declarations)
- `classifyDeps` — splits deps into "split-imports" (in our closure) vs "builtin-imports" (need `import Init`)

### Builtin Filtering (fixed in this session)
- `isBuiltin` uses **prefix matching** for module names:
  - `Init`, `Init.*` (e.g., `Init.Data.Bool`)
  - `Lean`, `Lean.*`
  - `Std`, `Std.*`
  - `Batteries`, `Batteries.*`
  - `Mathlib`, `Mathlib.*`
- Previous bug: used exact equality (`m == `Init``) which failed for hierarchical modules like `Init.Data.Bool`
- Fix: `mStr == "Init" || mStr.startsWith "Init."` etc.

### Auto-Generated Declaration Filtering (added in this session)
- Constructors (`.ctorInfo`) — skipped, part of parent inductive
- Recursors (`.recInfo`) — skipped, auto-generated
- Auto-generated definitions filtered by name suffix:
  - `.casesOn`, `.ctorElim`, `.ctorElimType`
  - `.below`, `.brecOn`
  - `.noConfusion`, `.noConfusionType`
  - `.recOn`, `.ctorIdx`
  - `.match_1` (and any `.match_N`)

### Inductive Extraction (works!)
- 8 inductive types compile standalone:
  - `TypeKind`, `RepresentationKind`, `TypeDefnKind`
  - `UnionReprStrategy`, `StructReprStrategy`
  - `MonsterObject`, `BottPhase`, `ContentId`
- Pretty-printed form (`inductive Foo where | bar : Foo`) is simple enough to round-trip

### Project Independence
- `lakefile.toml` has NO Mathlib dependency — split project is self-contained
- Only `import Init` is added when builtin deps are needed
- Split imports use `import Split.<decl_name>` format

### DAG Output
- `dag.json` records exact in-closure dependencies for each declaration
- Each declaration maps to an array of dependency names

## What Doesn't Work (The Core Problem)

### 172 out of 180 declarations FAIL to compile

All `def`, `theorem`, `instance`, and `opaque` declarations fail because the Lean
kernel stores fully-elaborated `Expr` terms that do NOT survive a pretty-print →
recompile round-trip.

### Specific Failure Modes

1. **Hygienic names**:
   ```
   x._@.RequestProject.IPLDMonsterSchema.3274500301._hygCtx._hyg.1
   ```
   These are internal hygienic variable names generated by the elaborator.
   They are not valid pattern variables in source syntax.

2. **Universe variables**:
   ```
   Sort.{u}
   ```
   Universe variables in pretty-printed output need explicit universe binders
   that the pretty-printer doesn't emit.

3. **Internal proof references**:
   ```
   IPLDMonsterSchema.instDecidableEqTypeKind._proof_1
   IPLDMonsterSchema.instDecidableEqTypeKind._proof_2
   ```
   These are separate declarations (proof terms) that are not in the split file.
   They're generated by the elaborator as auxiliary declarations.

4. **Auto-generated definitions** (casesOn, etc.):
   Even after filtering, some auto-generated defs slip through and use recursor
   references that don't compile standalone.

### Example: What a Failing File Looks Like

```lean
import Split.IPLDMonsterSchema_TypeKind

-- def IPLDMonsterSchema.instDecidableEqTypeKind : DecidableEq.{1} IPLDMonsterSchema.TypeKind
def IPLDMonsterSchema.instDecidableEqTypeKind : DecidableEq.{1} IPLDMonsterSchema.TypeKind :=
  fun (x._@.RequestProject.IPLDMonsterSchema.3274500301._hygCtx._hyg.1 : ...) => ...
```

The `x._@.RequestProject...` is not valid Lean source — it's a hygienic name
from the elaborator that only exists in the kernel's internal representation.

## The Fix: Source-Based Extraction

### Problem
The Lean kernel's `Expr` type stores fully-elaborated terms that are NOT designed
to be pretty-printed back to source syntax. The pretty-printer (`ToString Expr` /
`ppExpr`) produces output that looks like Lean source but isn't valid.

### Solution
Extract declaration bodies from the **original source files** instead of
pretty-printing from the kernel environment.

### Approach: "Extract the expr, find what lines produced it, capture those raw bytes"

1. **Kernel gives us**: exact dependency graph (which declarations depend on which)
2. **Source files give us**: compilable declaration text (the original source that the compiler parsed)
3. **Ship both**: kernel expr data in `dag.json` + raw source bytes in `.lean` files
4. **Repair into a decl**: prepend split imports + wrap in namespace

### API: `Lean.DeclarationRange`

The Lean environment stores source positions for each declaration via a
`PersistentEnvExtension` called `declRangeExt`.

**Module**: `Lean.DeclarationRange` (NOT `Lean.Elab.DeclarationRange`)

**Key types**:
```lean
structure Position where
  line   : Nat  -- 1-indexed
  column : Nat  -- 0-indexed

structure DeclarationRange where
  pos          : Position  -- start position
  charUtf16    : Nat       -- UTF-16 char count (for LSP)
  endPos       : Position  -- end position
  endCharUtf16 : Nat       -- UTF-16 end char count

structure DeclarationRanges where
  range          : DeclarationRange  -- full declaration range
  selectionRange : DeclarationRange  -- selection range (name position)
```

**Key function**:
```lean
-- In Lean.DeclarationRange module:
def findDeclarationRanges? [Monad m] [MonadEnv m] [MonadLiftT BaseIO m]
  (declName : Name) : m (Option DeclarationRanges)
```

**Problem**: `findDeclarationRanges?` requires `MonadEnv`, which we don't have in
the splitter (we use `IO` directly with an `Environment` value).

**Solution**: Use `declRangeExt.find?` directly with our environment:
```lean
-- declRangeExt is a MapDeclarationExtension DeclarationRanges
-- We can call find? directly with our Environment:
let ranges := Lean.declRangeExt.find? (level := .exported) env n <|>
              Lean.declRangeExt.find? (level := .server) env n
```

### Implementation Plan

1. **Add import**: `import Lean.DeclarationRange` (gives us `declRangeExt` and types)

2. **Add `findSourceFile` function**:
   - Convert module name to relative path: `RequestProject.IPLDMonsterSchema` → `RequestProject/IPLDMonsterSchema.lean`
   - Search in:
     a. Current working directory (for project modules)
     b. `.lake/packages/*/` directories (for dependency modules like Mathlib)
     c. Search path entries with `.lake/build/lib/lean` replaced (derived source dirs)

3. **Add `extractSourceText` function**:
   - Call `declRangeExt.find?` to get `DeclarationRanges`
   - Get the module name from `env.getModuleIdxFor?` + `env.header.moduleNames`
   - Find the source file via `findSourceFile`
   - Read the file, split into lines
   - Extract lines from `range.pos.line` to `range.endPos.line` (1-indexed)
   - Return the raw source text

4. **Add `getNamespace` function**:
   - Extract namespace from declaration name: `IPLDMonsterSchema.TypeKind` → namespace `IPLDMonsterSchema`, short name `TypeKind`
   - For declarations with no `.` in the name: no namespace needed

5. **Modify `emitDeclFile`**:
   - Try `extractSourceText` first
   - If successful: prepend split imports, wrap in `namespace <ns> ... end <ns>`, use raw source as body
   - If fails (no ranges or no source file): fall back to pretty-printed version (won't compile, but has dependency info)

6. **Add `sourceMap.json` output**:
   - Maps each declaration to its source file and line range
   - Useful for debugging and Aristotle verification

### Source File Structure (IPLDMonsterSchema.lean)

```
Line 1-37:  Module docstring (comment block)
Line 38:    namespace IPLDMonsterSchema
Line 46:    inductive TypeKind where
Line 52:    instance : Fintype TypeKind where
Line 59:    theorem type_kind_count : ...
...
Line 245:   end IPLDMonsterSchema
```

All declarations are inside `namespace IPLDMonsterSchema`. The source uses short
names (e.g., `TypeKind` not `IPLDMonsterSchema.TypeKind`). The namespace wrapper
in the split file will resolve these correctly.
